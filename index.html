Coleta de Evidências

Ordem de Volatilidade: Sempre colete dados na ordem do mais volátil para o menos volátil.

  1. Não desligue o servidor imediatamente (isso apaga a RAM).
  2. Documente tudo: Fotografe a tela e documente o estado físico antes de tocar na máquina.
  3. Coleta "Live" (Vivo): Se o servidor estiver ligado, colete a RAM primeiro.
  4. Coleta "Dead" (Morto): Se o servidor foi desligado, inicie a imagem do disco.
  5. Cadeia de Custódia: Mantenha um registro detalhado de quem, quando e como a evidência foi coletada
  6. Hashing: Após a coleta (especialmente de disco), gere um hash (SHA-256) para garantir a integridade da prova
  7. Tabelas de Roteamento, Cache ARP, Tabela de Processos.
  8. Sistemas de arquivos temporários
  9. Disco Rígido (Logs e arquivos)

1. CONTENÇÃO E PRESERVAÇÃO

Fluxo Detalhado de Contenção

1.0 INÍCIO - Disparador do processo de resposta a incidentes
1.1 Alerta de Comprometimento de Servidor

Recebimento de alerta do SIEM, EDR, usuário ou monitoramento proativo indicando possível comprometimento.

Verificação inicial: Confirmar se o alerta é legítimo (falso positivo?)

1.2 ISOLAMENTO LÓGICO IMEDIATO

Ação prioritária para conter a ameaça e evitar propagação:

  • Bloquear no firewall de borda (portas específicas ou todo o tráfego)
  • Isolar em VLAN/quarentena no switch
  • Regras de Host-Firewall (iptables/Windows Firewall com regras restritivas)
  • Manter acesso da equipe de resposta (SSH/RDP apenas de IPs específicos)

Objetivo: Impedir movimentação lateral e comunicação com C2 (Command & Control).

1.3 Capturar Memória RAM

Executar imediatamente após isolamento, antes de qualquer reinício:

SistemaFerramentaComando/Procedimento
WindowsFTK Imager Lite / DumpItExecutar como admin, salvar dump em mídia externa
LinuxAVML / LiMEsudo avml --output /mnt/externo/memory.dump
ESXivmss2coreExtrair memória da VM suspensa

Hash MD5/SHA256 do arquivo de dump deve ser registrado para cadeia de custódia.

1.4 Registro de Sessões

Documentar estado atual antes de qualquer alteração:

  • Conexões de rede: netstat -ano (Win) / ss -tulpn (Linux)
  • Processos em execução: tasklist /v (Win) / ps auxf (Linux)
  • Serviços: sc query (Win) / systemctl list-units (Linux)
  • Agendadores de tarefa: Agendador de Tarefas (Win) / Cron (Linux)
  • Usuários logados: query user (Win) / who -a (Linux)

Salvar saída em arquivo com timestamp: YYYY-MM-DD_HHMM_servidor_processos.txt

1.5 Monitoramento de Conexões

Configurar captura contínua para detectar tentativas de reconexão:

  • tcpdump/Wireshark: tcpdump -i any -w /mnt/externo/captura.pcap
  • NetFlow/sFlow: Habilitar exportação para coletor
  • EDR: Manter agente ativo para telemetria contínua

Isso permite identificar portas de backdoor e IPs de C2 ainda ativos.

Checklist de Preservação Forense

✓ Chain of Custody (Cadeia de Custódia)
Documentar TODAS as ações: Quem, quando, o que fez, com qual ferramenta.
✓ Hash de Evidências
Calcular MD5/SHA256 de todos os arquivos coletados antes de movê-los.
✓ Mídia Limpa
Usar mídia externa NOVA/formatada para armazenar evidências.
✓ Fuso Horário
Registrar timezone do servidor e usar UTC em logs da investigação.

2. ANÁLISE E DECISÃO

Fluxo de Análise e Decisão Crítica

2.1 Avaliar custo do bloqueio vs dano do malware

Análise de Impacto ao Negócio:

  • Servidor Crítico? (ERP, banco de dados, autenticação)
  • Disponibilidade vs Segurança: Quanto custa cada hora de downtime?
  • Horário do ataque: Horário comercial vs fim de semana
  • Contratos SLA: Penalidades por indisponibilidade

Decisão: Se custo do bloqueio < dano potencial → Prosseguir com contenção agressiva.

2.2 Risco de Propagação Imediata?

Indicadores de propagação:

  • Malware do tipo worm (ex: WannaCry, NotPetya)
  • Credenciais de domínio encontradas no servidor
  • Conexões ativas para outros servidores internos
  • Arquivos em shares de rede acessíveis

Ações de verificação:

TécnicaFerramenta/ComandoO que verificar
Busca por IOCsEDR Query, SIEM SearchMesmo hash/IP em outros sistemas
Logs de AutenticaçãoEvent ID 4624/4625 (Win)
auth.log (Linux)		Tentativas do servidor para outros sistemas
Scan de Rede InternaNmap, NessusPortas abertas anormais em outros hosts

Se RISCO ALTO de propagação:

→ Isolar imediatamente a rede/segmento afetado

→ Alertar toda a equipe de segurança

→ Considerar declarar incidente de segurança de nível 1

2.3 Contatar responsável pelo Servidor

Comunicação estruturada:

  • Proprietário do sistema: Quem gerencia a aplicação/serviço?
  • Equipe de infraestrutura: Virtualização, storage, backup
  • Compliance/Risco: Necessidades de regulatórias (LGPD, PCI DSS)

Informações a fornecer:

  1. Natureza do incidente (suspeita de comprometimento)
  2. Evidências encontradas até o momento
  3. Impacto operacional estimado
  4. Tempo estimado de indisponibilidade
  5. Plano de ação proposto
2.4 Aguardar resolução e feedback DSInf

Enquanto aguarda decisão do proprietário:

  • Continuar análise forense (análise de memória, logs)
  • Preparar plano de erradicação (se for mantido online)
  • Preparar imagem limpa para restore (se for mantido offline)
  • Documentar todas as descobertas para report

Prazo máximo: Estabelecer tempo limite para decisão (ex: 2 horas)

DECISÃO FINAL Árvore de Decisão: Manter Online vs Offline

MANTER OFFLINE (RECOMENDADO)

Quando escolher:

  • Evidência clara de comprometimento
  • Backdoor/Rootkit identificado
  • Servidor NÃO-CRÍTICO
  • Disponibilidade de backup recente
  • Risco regulatório alto (dados sensíveis)

Próximo passo: → Fase 3 (Erradicação com wipe/reinstall)

MANTER ONLINE (EXCEÇÃO)

Quando escolher:

  • Servidor CRÍTICO (dowtime inaceitável)
  • Comprometimento limitado/contido
  • Capacidade de remover malware sem reboot
  • Monitoramento avançado implementado
  • Aceitação formal de risco pela diretoria

Próximo passo: → Remoção dirigida + monitoramento intensivo

Backdoor com C2 ativoOFFLINEInterromper exfiltração de dados
